## 硅基一号逐日数据接入 PCSE 可行性分析

### 数据映射：PCSE 天气字段 ↔ 硅基一号传感器

PCSE 的 `WeatherDataContainer` 要求 12 个逐日字段。下面是每个字段与硅基一号 `sensor_readings` 表的对应关系：

| PCSE 字段 | 单位 | 硅基一号数据源 | 可用性 | 说明 |
|-----------|------|---------------|--------|------|
| `DAY` | date | `sensor_readings.timestamp` | ✅ 直接可用 | 按日期聚合 |
| `LAT` | ° | `device` 表 / 固定配置 | ✅ 直接可用 | 温室坐标 |
| `LON` | ° | `device` 表 / 固定配置 | ✅ 直接可用 | 温室坐标 |
| `ELEV` | m | 固定配置 | ✅ 直接可用 | 海拔 |
| `TMAX` | °C | `MAX(air_temp)` | ✅ 直接可用 | 日内最高气温 |
| `TMIN` | °C | `MIN(air_temp)` | ✅ 直接可用 | 日内最低气温 |
| `IRRAD` | J/m²/day | `solar_rad × 86400` | ✅ 可换算 | 瞬时 W/m² → 日累计 J/m² |
| `VAP` | hPa | 从 `air_temp` + `air_humidity` 推导 | ⚠️ 可推导 | 用饱和水汽压公式 |
| `RAIN` | cm/day | — | ❌ 温室无降雨 | 温室内恒为 0 |
| `WIND` | m/s | — | ❌ 温室无风 | 温室内≈0，无风速传感器 |
| `E0` | cm/day | — | ❌ 不适用 | 开放水面蒸发，温室无意义 |
| `ES0` | cm/day | — | ❌ 不适用 | 裸土蒸发，无土栽培无意义 |
| `ET0` | cm/day | — | ❌ 不适用 | Penman-Monteith 参考蒸散，大田公式 |

### 三层评估

```
           ✅ 可用的 (~30%)                ⚠️ 跑不准的 (~40%)              ❌ 根本用不了的 (~30%)
  ┌──────────────────────────────┐  ┌──────────────────────────────┐  ┌──────────────────────────────┐
  │ · 温度 → GDD 积温驱动物候     │  │ · 水分平衡 → 温室无雨无风      │  │ · 番茄无限生长 vs DVS=2 死亡  │
  │ · 辐射 → 光合/干物质积累      │  │   PCSE 会误判为持续干旱        │  │ · 基质含水量 vs 土壤水分       │
  │ · 日长 → 可以正常计算         │  │ · CO₂ 富集 → 温室 800-1200 ppm │  │ · 环控反馈 → PCSE 无此模型     │
  │                              │  │   PCSE 假设大气 ~400 ppm       │  │ · 营养液配方切换 → 无对应模块  │
  │                              │  │ · LED 补光 → 与自然光混在一起   │  │ · 无限生长型采收轮次 → 无法表达 │
  │                              │  │   PCSE 无法区分光源            │  │                              │
  └──────────────────────────────┘  └──────────────────────────────┘  └──────────────────────────────┘
```

### 详细分析

#### ✅ 可用的部分

**1. 温度 → GDD 积温驱动物候**

硅基一号已有完整的 GDD 积温计算模块 (`gdd_tracking` 表)。空气温度是温室内最可靠的传感器数据，PCSE 的温度驱动物候逻辑 (TSUM1/TSUM2) 可以直接使用：

- `TMAX` = 当日 `MAX(air_temp)`
- `TMIN` = 当日 `MIN(air_temp)`
- GDD = `(TMAX + TMIN) / 2 - TBASE`（与 PCSE 公式一致）

**2. 辐射 → 光合作用与干物质积累**

硅基一号有 `solar_rad` (W/m²) 传感器，单位换算即可：

```
IRRAD (J/m²/day) = solar_rad_avg × 86400
```

这是 PCSE 光合作用模型的核心驱动变量，直接影响同化物生产和生物量积累。

#### ⚠️ 跑不准的部分

**1. 水分平衡 — 最严重的偏差源**

PCSE 的水分平衡基于大田 Penman-Monteith 公式：

```
ET0 = f(辐射, 温度, 风速, 水汽压)
```

温室内 `RAIN=0`、`WIND≈0`、湿度受控，导致：
- PCSE 计算出极低的蒸散量（无风 → 低 ET0）
- 无降雨输入 → 土壤水分只能下降
- 模型会持续触发水分胁迫 → `RFTRA` 远低于 1.0
- 模拟结果偏差可超过 40%

**2. CO₂ 富集效应缺失**

温室内 CO₂ 浓度通常 800-1200 ppm（补充），远高于大田大气 ~400 ppm。PCSE 的光合作用模型 (`AMAXTB`) 假设大气 CO₂ 浓度，无法体现温室 CO₂ 富集带来的光合增益（通常 +20-40% 产量）。

**3. LED 补光无法区分**

硅基一号有 `led_par` 传感器。但 PCSE 的辐射输入是单一 `IRRAD` 值，无法区分自然光和补光的光谱差异（LED 红蓝光光合效率更高）。补光时长和强度的增产效应会被低估。

#### ❌ 根本用不了的部分

**1. 番茄无限生长 vs WOFOST DVS 0→2 生命周期**

这是最根本的模型不兼容：

```
WOFOST (大田一年生):  DVS 0 (出苗) → 1 (开花) → 2 (成熟/死亡)
温室番茄 (无限生长):  DVS 0 (定植) → ... 持续营养+生殖生长，无自然死亡点
```

WOFOST 到达 DVS=2 后停止模拟，干物质分配切换到"全部去储存器官"。温室番茄则持续同时进行营养生长和生殖生长，没有"全部灌浆"阶段。

**2. 基质栽培 vs 土壤模型**

PCSE 的土壤水分模块假设田间土壤剖面（毛细管上升、深层渗漏、地表径流）。温室基质栽培（椰糠/岩棉）的物理特性完全不同：

| 特性 | 大田土壤 | 温室基质 |
|------|---------|---------|
| 持水曲线 | 宽范围的 PF 曲线 | 陡峭的释放曲线 |
| 导水率 | 随含水量渐变 | 饱和时极高，稍干骤降 |
| 根区深度 | 60-150 cm | 10-20 cm (基质袋) |
| 灌溉方式 | 降雨 + 漫灌 | 高频滴灌 (每天数次) |

强行套用土壤参数会导致水分胁迫计算完全失真。

**3. 环控系统反馈闭环缺失**

PCSE 是开环模型（天气输入 → 作物输出），而温室是闭环系统：

```
传感器 → 环控决策 → 执行器动作 → 环境改变 → 传感器
```

PCSE 无法模拟"温度过高→开启通风→温度下降"或"光照不足→开启补光灯→PAR 增加"的反馈循环。

### 接入方案：最小可行路径

如果仍然希望在 PCSE 中运行硅基一号数据（例如用于 GDD 验证或作为 Semantica 决策的参考输入），推荐以下最小改动方案：

```
1. 自定义 WeatherDataProvider 子类
   └── class Silicon1WeatherProvider(WiNPyWeatherDataProvider):
       从 sensor_readings 表按日期聚合

2. 缺失字段的合理填充
   ├── RAIN = 0 (温室无降雨)
   ├── WIND = 0.5 (微小值防除零)
   ├── E0/ES0/ET0 = 0 或从 IRRAD+T+RH 估算
   └── VAP = 从 air_temp + air_humidity 计算

3. 不修改 PCSE 引擎核心
   └── 仅在数据输入层做适配，不做 WOFOST 模型改动

4. 结果解释时标注限制
   ├── 水分胁迫指标 (RFTRA, SM) 无参考意义
   ├── 生物量/产量预测偏低 (缺 CO₂ + 补光增益)
   └── DVS 到达 2 后模拟停止，不代表番茄实际死亡
```

### 结论

硅基一号的温湿度/辐射数据**可以喂给 PCSE**，但模拟结果只能作为**有限参考**：
- **温度驱动的 GDD 积温和物候阶段判断**是唯一比较可靠的部分
- **生物量/产量预测**会系统性偏低（缺乏 CO₂ 和补光增益的补偿）
- **水分胁迫指标**在温室场景下完全不可用
- **番茄无限生长特性**与 WOFOST 的一年生模型根本矛盾

如果硅基一号需要一个作物生长模拟引擎，长期方向应该是实现一个温室专用的简化模型（基于 GDD + 光截获 + CO₂ 响应），而不是强行适配大田模型。

## 场景修正：如果硅基一号是大棚而非全控温室

上述分析基于**全控温室**（玻璃/PC板温室，CO₂ 补充、LED 补光、基质栽培、环控系统完备）。如果硅基一号的应用场景是**大棚**（塑料大棚/冷棚/拱棚），情况会大幅度改善。

### 大棚 vs 全控温室的关键差异

| 维度 | 全控温室 | 大棚 | 对 PCSE 的影响 |
|------|---------|------|---------------|
| **温度** | 主动加温，可维持 20-28°C | 被动保温，比外界高 2-5°C | 大棚温度更接近自然波动，适合 PCSE |
| **土壤** | 椰糠/岩棉基质袋 | **天然土壤** | ✅ 大棚用土，PCSE 土壤模型直接适用 |
| **降雨** | 完全隔离 | 完全隔离 | 仍然 RAIN=0，但滴灌可映射为 RAIN |
| **风速** | ≈0 | 卷膜通风，有自然对流 | 大棚有微弱风，ET0 计算不至于失真 |
| **CO₂** | 800-1200 ppm (补充) | ~400 ppm (大气水平) | ✅ 大棚无富集，与 PCSE 假设一致 |
| **补光** | LED 补光 (PAR 增加) | 无补光，仅自然光 | ✅ 单一自然光源，PCSE 可直接使用 |
| **环控** | 主动调控（加热/通风/遮阳/补光） | 被动管理（手动卷膜/放风） | ✅ 大棚近似开环，与 PCSE 一致 |
| **栽培方式** | 无限生长型 (1-2 季/年) | 通常一年一茬或两茬 | 更接近一年生模式 |

### 大棚场景下的可行性重新评估


### 大棚特定的适配建议

**1. 塑料膜透光率修正**

大棚膜（PE/PVC/PO 膜）会削减 PAR，需在辐射换算中加入透光率：

```
IRRAD = solar_rad_avg × 86400 × τ

τ = 0.70 (旧膜/落尘) ~ 0.90 (新膜/清洁)
```

如果不修正，PCSE 会高估光合有效辐射，导致生物量预测偏高。

**2. 灌溉 → RAIN 映射**

大棚无天然降雨，但滴灌/沟灌可等效为"人工降雨"：

```yaml
# 在农事操作中，将灌溉表达为 TimedEvents
TimedEvents:
- event_signal: irrigate
  events_table:
  - 2024-03-15: {amount: 2.0, efficiency: 0.9}
  - 2024-03-18: {amount: 2.0, efficiency: 0.9}
```

或者更简单：直接在 WeatherDataProvider 中将灌溉量填入 `RAIN` 字段。两种方式等价，但农事表达更透明（可追踪每次灌溉的量和效率）。

**3. 风速处理**

大棚内有微弱空气流动（卷膜通风、热浮力对流），建议 WIND 设为 0.3-0.8 m/s（而非全控温室的 ≈0）：

```
WIND = 0.5 m/s  (大棚默认值)
```

这样 Penman-Monteith 的 ET0 计算不会完全失真，可以给出合理数量级的蒸散量。

**4. 番茄品种选择**

大棚番茄虽然也是无限生长型，但实际生产中通常：
- 秋季定植 → 冬季/春季采收 → 夏季拉秧（6-8 个月）
- 留 6-8 穗果后打顶（人工封顶）

这意味着大棚番茄**客观上存在生长终点**，与 WOFOST 的 DVS 生命周期可以近似对齐：

```
大棚番茄 ≈ 长生育期一年生作物
TSUM1 (定植→开花):    600-800 °C·d  (约 30-45 天)
TSUM2 (开花→拉秧):   1000-1400 °C·d (约 60-90 天)
```

通过调整 `TSUM1`/`TSUM2` 并设置 `DVSEND=2.0`，可以让 WOFOST 大致覆盖大棚番茄的主要生育期。

### 大棚场景结论

| 评估维度 | 全控温室 | 大棚 |
|----------|---------|------|
| 整体可用度 | ~30% | ~60% |
| 温度/GDD | ✅ | ✅ |
| 辐射/光合 | ⚠️ 区分不了补光 | ✅ 纯自然光，加透光率修正 |
| 土壤水分 | ❌ 基质不兼容 | ✅ 天然土壤兼容 |
| CO₂ | ❌ 富集未建模 | ✅ 大气水平匹配 |
| 作物生命周期 | ❌ 无限生长 | ⚠️ 可近似为长生育期一年生 |
| 环控反馈 | ❌ 闭环模型缺失 | ✅ 近似开环 |
| 风速/ET | ❌ 无风失真 | ⚠️ 需设小值 |

**大棚场景下，PCSE 的可用度从 ~30% 提升到 ~60%**。最关键的两个变化是：(1) 土壤模型的兼容性恢复，(2) CO₂/补光/环控这些"多余能力"在大棚中本来就没有，不再是 PCSE 的缺陷。

剩余 ~40% 的不确定性主要来自：
- 塑料膜的辐射/温度缓冲效应（可参数化但需要实测校准）
- 番茄无限生长特性的截断近似（可以接受，因为实际生产中有打顶/拉秧）
- 灌溉与降雨的语义差异（技术层面可解决，用农事操作或 RAIN 字段替代）

```

    (15 分钟/次或按需上报)

②  硅基一号 Agent ──→ DataHub ──→ IoT Lite ──MQTT──→ 执行器
    (控制指令: 灌溉/补光/通风/加热)

③  硅基一号 Agent ──→ LLM 大模型
    (推理请求: 异常诊断/方案生成/知识问答)
    LLM ──→ Agent (推理结果 + 建议)

④  硅基一号 Agent ──→ Semantica 知识图谱
    (存储: 传感器快照 / 决策记录 / 事件日志)

    (查询: 历史相似场景 / 因果关系 / 经验规则)
```

### 逐层可行性评估

#### ① MQTT + IoT Lite：✅ 完全可行

这是最成熟的环节。MQTT 是物联网事实标准协议：
- **QoS 三级**：QoS 0/1/2 可适配传感器数据（可丢失）和控制指令（必须到达）
- **低带宽**：二进制协议，4G/NB-IoT 弱网可用
- **设备影子**：离线设备重连后同步最新期望状态
- **成熟方案**：EMQX / Mosquitto / 阿里云 IoT / 华为 IoT 均可

建议关注的点：
- 控制指令必须 QoS 1 起步，命令去重（MQTT messageId）
- 设备侧做指令有效期校验（超过 N 秒的旧指令丢弃）

#### ② DataHub 数据总线：✅ 可行，但需要明确边界

DataHub 在这一层承担三个角色：

| 角色 | 说明 | 风险 |
|------|------|------|
| 协议转换 | MQTT ↔ HTTP/gRPC/消息队列 | 低风险，成熟模式 |
| 数据清洗 | 异常值过滤、单位换算、时间戳对齐 | 中风险，规则需持续维护 |
| 消息路由 | 传感器数据 → Agent，指令 → IoT | 低风险 |

**关键问题**：DataHub 是自研还是复用？建议优先评估现有方案（Apache Kafka + Kafka Connect、EMQX Rule Engine、Node-RED），避免在这个环节自研造轮子。

#### ③ LLM 大模型连接：⚠️ 可行，"自学习"需要澄清

这里需要区分三个不同层次的能力：

```
L1 — 推理规划 (可行)
  Agent 将传感器数据 + 历史记录打包为 prompt，LLM 输出诊断/建议
  实现: prompt engineering + function calling
  延迟: 2-10 秒 (非实时场景可接受)

L2 — RAG 增强 (可行)
  从 Semantica 检索历史相似场景，注入 LLM context
  实现: embedding + 向量检索 + 知识图谱子图提取
  效果: 让 LLM "记住"过去的经验

L3 — 在线自学习 (困难)
  LLM 在部署后持续从新数据中更新参数
  实际上很难做到: 需要 RLHF/微调 pipeline + 人工标注 + 安全护栏
```

**务实建议**：L1 + L2 即可覆盖绝大多数场景。"自学习"在短期应该理解为**RAG + few-shot + 人工反馈写入 Semantica 知识图谱**，而非模型权重更新。

#### ④ Semantica 知识图谱：✅ 可行，与 Semantica 的现有能力高度匹配

Semantica 已有的四模块直接对应需求：

| Semantica 模块 | 对应需求 | 适配度 |
|---------------|---------|--------|
| Context Graph | 传感器数据 → 实体关系（设备-地块-茬次-品种） | 天然匹配 |
| Decision Intelligence | 决策追踪、因果链（P→S→W→D→E→V） | 天然匹配 |
| Provenance (W3C PROV-O) | 每个控制动作可追溯到触发原因 | 天然匹配 |
| Reasoning Engine | SPARQL/Datalog 查询历史模式辅助决策 | 天然匹配 |

**数据流向**：
```
硅基一号 → Semantica: 传感器快照、控制动作、决策记录、事件日志
Semantica → 硅基一号: 相似历史场景、因果链查询、经验规则匹配
```

#### ⑤ PCSE 生成种植方案：✅ 可行（大棚场景）

PCSE 可以为每一季生成初始种植方案，包括：

```
方案输出:
├── 品种选择:      基于目标产量和棚内历史积温选择
├── 定植日期:      基于目标采收期倒推
├── 预期生育阶段:   TSUM1 开花日 / TSUM2 拉秧日
├── 目标产量:       TAGP / TWSO 基于历史天气
├── 灌溉计划:       基于土壤参数和预期 ET0
└── 风险提示:       基于历史极端天气概率
```

#### ⑥ PCSE 长势预测：✅ 可行（大棚场景，需校准）

用硅基一号的真实逐日数据驱动 PCSE，输出：
- 当前 DVS 及距离开花/拉秧天数
- LAI 和生物量曲线与预期的偏差
- 胁迫因子 (RFTRA) 趋势预警

**校准闭环**：
```
PCSE 预测值 → 对比硅基一号实测值 (LAI/生物量/产量) → 修正 PCSE 参数 → 下次预测更准
```

### 建议优先实现路径

```
Phase 1 (基础闭环):
  MQTT设备 → IoT Lite → DataHub → 硅基一号 → DataHub → IoT Lite → MQTT → 设备控制
  目标: 传感器数据能流入，控制指令能下发

Phase 2 (知识沉淀):
  硅基一号 → Semantica 知识图谱 (传感器快照 + 控制记录)
  目标: 所有决策可追溯

Phase 3 (AI 增强):
  硅基一号 ← → LLM (诊断 + 建议)
  硅基一号 ← → Semantica (历史检索 + RAG)
  目标: AI 辅助决策

Phase 4 (模拟驱动):
  PCSE 生成初始种植方案
  PCSE 逐日长势预测 + 偏差预警
  目标: 数据驱动的精准种植

Phase 5 (持续优化):
  实际产量 vs PCSE 预测 → 参数校准
  LLM 决策效果反馈 → 经验规则沉淀 → Semantica
  目标: 越种越准
```

### 最大的三个风险点

1. **"自学习"的预期管理**：LLM 不会自己变聪明。需要在 Semantica 中沉淀好的决策 + 人工标注的效果反馈，用 RAG 让 LLM "看起来"在学习。真正的模型微调需要大量标注数据和工程化 pipeline。

2. **PCSE 的精度期望**：在大棚场景下 PCSE 可以用，但它本质是逐日步长的机理模型，不能期望对短时间尺度（小时级）的环境变化做出响应。产量预测误差 15-30% 是正常范围。

3. **DataHub 的单点风险**：DataHub 是整条链路的咽喉。需要考虑：宕机时设备能否本地自治？MQTT 消息能否缓存重放？Agent 有无降级策略（直接读 IoT Lite、跳过 DataHub）？
