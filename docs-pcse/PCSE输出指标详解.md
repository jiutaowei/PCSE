## 1. 产量 TWSO — Total Weight Storage Organs

| 属性 | 说明 |
|------|------|
| 全称 | Total Weight Storage Organs |
| 单位 | kg/ha |
| 含义 | **储存器官总干重，即经济产量** |

**逐作物对应关系**：

| 作物类型 | TWSO 对应 |
|----------|----------|
| 谷类（小麦、大麦、玉米、水稻） | 籽粒产量 |
| 根茎类（马铃薯、甜菜） | 块茎/块根产量 |
| 油料（油菜、向日葵） | 种子产量 |
| 牧草（饲料玉米、牧草） | 不适用（crop_model=1 时关注 TAGP） |

**解读**：种植方案里最重要的经济指标——最终能收获多少可售卖的部分。7180 kg/ha 约合每亩 479 公斤小麦籽粒。TWSO=0 通常意味着作物在该气候条件下无法完成生殖生长（如表中玉米在 Sevilla 冬季不抽穗）。

**冬小麦品种级差异**：数据库 50 个小麦品种的 TSUM1 从 518 到 852°C·d 不等，意味着不同品种在同一地区的 TWSO 可能有 20%-40% 的差异。品种选择就是通过逐品种模拟找到 TWSO 最大值。

---

## 2. 总生物量 TAGP — Total Above Ground Production

| 属性 | 说明 |
|------|------|
| 全称 | Total Above Ground Production |
| 单位 | kg/ha |
| 含义 | **地上部分所有器官干重总和**（叶片 + 茎秆 + 储存器官） |

**组成部分**：
```
TAGP = TWLV (叶片) + TWST (茎秆) + TWSO (籽粒/储存器官)
```

**解读**：反映光合作用的总产出，即太阳辐射能转化为化学能的效率。冬小麦 15262 kg/ha 意味着全生育期每公顷固定了约 15 吨干物质。TAGP 本身不代表经济价值——如果大部分去了秸秆而不是籽粒，就是"只长秆子不结粮"。

**TAGP 高但 TWSO 低的原因**：
- 生殖生长期遭遇严重水分胁迫 → 灌浆受阻
- 光周期不匹配 → 无法触发开花
- 品种积温需求与当地热量资源不匹配 → 生育期提前或延后

## 3. 收获指数 HI — Harvest Index

| 属性 | 说明 |
|------|------|
| 全称 | Harvest Index |
| 计算 | **HI = TWSO / TAGP** |
| 范围 | 0 到 1，常见 0.30-0.55 |

**含义**：光合产物中有多大比例最终分配到了经济器官。

**判断标准**：

| HI 值 | 解读 | 表中案例 |
|-------|------|---------|
| 0.50+ | 优秀，现代高产品种的育种目标 | 马铃薯 0.50 |
| 0.45-0.50 | 正常，符合现代品种特征 | 冬小麦 0.47 |
| 0.30-0.40 | 偏低，干物质分配偏向茎叶 | 春大麦 0.33 |
| 0.10-0.25 | 严重偏斜，生殖生长受阻 | 冬油菜 0.16 |
| <0.10 | 作物与环境完全不匹配 | 向日葵 0.06, 玉米 0.00 |

**为什么冬油菜 HI 只有 0.16？** 油菜的收获指数天然偏低（油脂合成耗能高），加上 Sevilla 冬季低温可能影响了开花后的种子充实过程。但 2509 kg/ha 在冬油菜中仍属可接受水平。

**为什么玉米 HI=0？** 玉米是短日照 C4 作物，在 Sevilla 冬春季低温（最低温常低于 10°C）和日长条件下无法正常抽穗吐丝，TWSO=0 导致 HI=0。

## 4. 生育期 — Growing Duration

| 属性 | 说明 |
|------|------|
| 单位 | 天 |
| 起点 | 出苗日 (emergence) |
| 终点 | 生理成熟日 (maturity, DVS=2.0) |

**决定因素**：

```
生育期天数 = f(品种积温需求 TSUM1+TSUM2, 逐日有效温度)
```

- **TSUM1**：DVS 0→1（出苗→开花）所需积温，冬小麦品种间差异 518-852°C·d
- **TSUM2**：DVS 1→2（开花→成熟）所需积温，冬小麦品种间差异 481-877°C·d
- **逐日有效温度** = max(0, Tavg - TBASE)，冬季低温时有效积温日增量很小

**解读**：

| 作物 | 生育期 | 日均干物质积累 | 解读 |
|------|--------|---------------|------|
| 马铃薯 | 97天 | 126 kg/ha/天 | 短周期高效率，日均积累最高 |
| 春大麦 | 122天 | 118 kg/ha/天 | 中等周期，适合同年轮作 |
| 冬小麦 | 152天 | 100 kg/ha/天 | 长周期积累多但日均效率中等 |
| 冬油菜 | 161天 | 98 kg/ha/天 | 最长周期，日均积累偏低 |

天数短不等于差——马铃薯 97 天完成全周期且产量 6137 kg/ha，日均干物质积累率反而是最高的。天数的意义在于与轮作日历的兼容性：冬小麦 152 天到 5 月底收获，正好可以接一茬夏玉米或蔬菜。

## 5. 水分胁迫天数 — Water Stress Days

| 属性 | 说明 |
|------|------|
| 判断标准 | **RFTRA < 0.95 的天数** |
| RFTRA 含义 | 实际蒸腾量 / 潜在蒸腾量 |

**RFTRA (Reduction Factor for Transpiration)** 是水分胁迫的直接量化指标：

| RFTRA 值 | 胁迫程度 | 生理含义 |
|----------|---------|---------|
| 1.0 | 无胁迫 | 气孔全开，蒸腾和光合均正常 |
| 0.95-0.99 | 轻微胁迫 | 土壤水分开始低于田间持水量 |
| 0.8-0.95 | 中度胁迫 | 气孔部分关闭，光合速率开始下降 |
| 0.5-0.8 | 重度胁迫 | 光合大幅降低，生长明显减缓 |
| <0.5 | 极端胁迫 | 接近凋萎点，植物濒临死亡 |

**表中胁迫天数的含义**：

| 作物 | 胁迫天数 | 占生育期% | 解读 |
|------|---------|----------|------|
| 春大麦 | 40天 | 33% | 胁迫程度最低，耐旱性相对好 |
| 冬小麦 | 42天 | 28% | 春季升温后蒸发加快导致 |
| 冬油菜 | 43天 | 27% | 长生育期但胁迫比例低 |
| 马铃薯 | 46天 | 47% | 短周期但胁迫密集，需灌溉 |
| 向日葵 | 74天 | 60% | 大部分时间缺水 |
| 玉米 | 102天 | 76% | 几乎全程胁迫 + 低温叠加 |

胁迫天数高不是绝对的"坏"——种子只要能完成关键生育期即可。但胁迫天数占比 > 50% 通常意味着需要考虑补灌方案或选择更抗旱的品种。

## 6. 六指标之间的关系

```
                    太阳辐射 + CO₂ + 温度
                          │
                          ▼
                    ┌──────────┐
                    │  光合作用  │
                    └────┬─────┘
                         │
                         ▼
               ┌─────────────────┐
               │ 总生物量 (TAGP)  │ ◄── 受水分胁迫 (RFTRA) 抑制
               └────┬────┬───┬───┘
                    │    │   │
          ┌─────────┘    │   └─────────┐
          ▼              ▼             ▼
       叶片(TWLV)    茎秆(TWST)    籽粒(TWSO)
                                       │
                                       ▼
                              收获指数 HI = TWSO/TAGP

    温度 ──→ 发育速度 ──→ 生育期天数 ──→ 水分需求总量
                                            │
    降雨 + 土壤水分 ──→ RFTRA ──→ 胁迫天数 ──→ TAGP ↓ & HI ↓
```

**逻辑链**：

1. **TAGP 告诉你地力行不行**——光照、温度、CO₂ 这些基础资源转化了多少干物质
2. **HI 告诉你品种行不行**——干物质分配策略是否把资源导向了收获器官
3. **胁迫天数告诉你管理行不行**——水分缺口多大，是否需要补灌
4. **TWSO = TAGP × HI**——产量是资源利用效率 × 分配效率的乘积
5. **生育期告诉你茬口衔接**——收获后地块何时空出来接下一季

## 8. 从空白地块到种植方案 — 自动生成器

上面的六作物对比需要手动逐作物模拟。`auto_planting_plan.py` 把整个过程自动化了：**输入经纬度和土壤基础参数，自动输出种植方案报告**。

### 用法

```bash
cd D:\work\code\silicon1\pcse

# 默认参数（Sevilla, 2023年, 水限模式）
python doc/downloads/auto_planting_plan.py

# 指定位置
python doc/downloads/auto_planting_plan.py --lat 39.9 --lon 116.4 --year 2023

# 指定初始有效水分（干旱条件）
python doc/downloads/auto_planting_plan.py --lat 37.64 --lon -6.09 --wav 5

# 潜在生产模式（假设水分无限，仅看光温潜力）
python doc/downloads/auto_planting_plan.py --lat 37.64 --lon -6.09 --mode pp
```

### 参数说明

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--lat` | 37.64 | 纬度（正值=北纬） |
| `--lon` | -6.09 | 经度（正值=东经） |
| `--year` | 2023 | 模拟年份（决定使用的气象数据） |
| `--wav` | 10 | 初始有效水分（cm），播种时土壤剖面的有效含水量 |
| `--mode` | wlp | 生产模式：`wlp`=水限（模拟实际水分条件），`pp`=潜在（仅受光温限制） |

### 工作流程

```
用户输入 (lat, lon, year, wav)
       │
       ├── [1] NASA Power API ──→ 逐日天气 (Tmax, Tmin, 辐射, 降雨, 风速, 水汽压)
       │
       ├── [2] PCSE Demo DB ──→ 6 种作物的生理参数 (TSUM1/2, AMAXTB, SLATB 等)
       │         用户指定 ──→ 土壤参数 (田间持水量, 萎蔫点, 根区深度)
       │         用户指定 ──→ 场地参数 (初始有效水分 WAV)
       │
       ├── [3] 逐作物运行 WOFOST 水限模型 ──→ 输出逐日 DVS, LAI, TAGP, TWSO, RFTRA
       │
       └── [4] 提取关键指标 ──→ 排序 ──→ 推荐方案 + 备选方案
```

### 实际运行示例

以 Sevilla (lat=37.64, lon=-6.09) 2023 年水限模式为例：

```
┌────────────────┬──────────┬──────────┬──────┬──────┬──────────┐
│ 作物           │ 产量     │ 生物量   │  HI  │ 天数 │ 胁迫天   │
├────────────────┼──────────┼──────────┼──────┼──────┼──────────┤
│ WINTER WHEAT   │    7,490 │   17,475 │ 0.43 │  169 │     57天 │  ← 推荐
│ POTATO         │    5,219 │   13,246 │ 0.39 │  108 │     43天 │
│ SPRING BARLEY  │    2,459 │   11,794 │ 0.21 │  132 │     49天 │
│ WINTER RAPESEED│    2,092 │   19,840 │ 0.11 │  182 │     60天 │
│ SUNFLOWER      │       43 │    2,095 │ 0.02 │  130 │     79天 │
│ GRAIN MAIZE    │        0 │    4,905 │  --  │  141 │    109天 │
└────────────────┴──────────┴──────────┴──────┴──────┴──────────┘
```

**推荐方案**：冬小麦，预期产量 7,490 kg/ha，出苗 1月1日，开花 4月17日，成熟 6月17日，全季 169 天。

### 数据来源

| 数据 | 来源 | 限制 |
|------|------|------|
| 天气 | NASA Power API (在线) | 0.5° 分辨率，全球覆盖，卫星反演 |
| 作物参数 | PCSE Demo DB (本地) | 仅 6 种欧洲大田作物，品种为默认品种 |
| 土壤参数 | 用户指定 + 默认值 | 默认为通用壤土，可用 `--wav` 调整水分 |
| 场地参数 | 用户指定 + 默认值 | 径流、地表蓄水等暂用零值 |

### 能做什么与不能做什么

**能做的**：
- 输入任意全球坐标，自动获取天气数据跑模拟
- 6 种作物横向对比，按产量排序
- 输出关键物候期（出苗、开花、成熟日期）
- 评估水分胁迫程度，判断是否需要灌溉支撑
- 切换 pp/wlp 模式对比光温潜力与水限实际

**不能做的**：

- 品种级精细筛选（数据库有 50 个小麦品种但脚本只用默认品种，需改代码遍历品种）
- 播种期优化（目前固定用 demo DB 的作物日历日期，需改 `crop_start_date` 跑多次对比）
- 经济分析（无市场价格、成本数据）
- 施肥方案（WOFOST 无养分模块）
- 温室/设施农业（模型假设开放大田）

**具体操作**：

1. 输入：经度、纬度、年份、初始有效水分（--lat --lon --year --wav）
  2. 自动拉取天气：NASA Power API，不需要本地气象站数据
  3. 自动构建参数：从 PCSE Demo DB 读取 6 种作物的生理参数，土壤用通用壤土默认值
  4. 逐作物模拟：跑一遍水限模型，记录产量、生物量、HI、生育期、胁迫天数
  5. 排序输出：按产量从高到低，给出推荐方案 + 备选方案

  用不同参数试试：
  # 北京附近
  python doc/downloads/auto_planting_plan.py --lat 39.9 --lon 116.4 --year 2023

  # 更干旱的条件
  python doc/downloads/auto_planting_plan.py --lat 37.64 --lon -6.09 --wav 5

  # 潜在生产模式（假设水不是限制）
  python doc/downloads/auto_planting_plan.py --lat 37.64 --lon -6.09 --mode pp

  但诚实地说局限：这仍然是 PCSE Demo DB 的 6 种欧洲大田作物，土壤参数是通用默认值而不是实测值，品种也只用了默认品种（未遍历 50
  个品种做精细筛选）。它是一个"基于物理模型的初筛工具"，不能替代农艺师的经验判断。





### PCSE定位

PCSE 是一个作物生长的物理模拟引擎——你给它天气、土壤、品种参数，它逐日计算作物从出苗到成熟的完整生命周期。

---
  它不是数据库查询系统，不是决策支持系统，更不是种植方案生成器。它的核心就一件事：

  输入                              输出
  ────────────────────────────────────────────
  天气 (每天温度/降雨/辐射)         逐日状态 (DVS/LAI/生物量/产量)
  土壤 (持水量/导水率/根区深度)  →  最终产量 (籽粒/块茎 kg/ha)
  品种 (积温需求/光合参数/分配系数)  水分胁迫 (RFTRA 曲线)
  管理 (播期/灌溉/收获策略)         物候期 (开花日/成熟日)

  本质上它做的事：给定以上输入，求解一组描述作物生长的微分方程，逐日步进，模拟整个生长季。跟天气预报模型跑大气物理方程是一个思路，只不过这里跑
  的是光合作用、呼吸作用、干物质分配、土壤水分平衡等农学方程。

---
  它的定位

  PCSE 是 Wageningen 大学为 欧盟作物监测业务系统 (CGMS/MARS) 开发的计算内核，不是给个体农户用的工具。它的典型用户场景是：

  - 欧盟农业部门每十天跑一次全欧 10000+ 网格点的模拟，出产量预报报告
  - 研究人员修改作物参数，模拟气候变化对小麦产量的影响
  - 育种公司对比不同品种的积温需求在目标区域的表现

  它不负责（也不该负责）：
  - 告诉你种什么赚钱——那是农艺师+市场分析的事
  - 告诉你怎么施肥——WOFOST 模型本身没有养分模块
  - 实时控制温室设备——它是离线批处理，不是边云协同推理

  │                           可复用                            │                不可复用                 │

## 9. 农事操作 (Agromanagement)

农事操作是 PCSE 里控制"人什么时候做什么"的模块。模型负责算作物怎么长，农事管理负责告诉模型"今天该播种了""今天该施肥了""今天该收了"。

### 9.1 三部分结构

每个生长季 (Campaign) 的定义由三部分组成，写在一个 YAML 文件里：

```
一个 Campaign (活动周期)
  ├── CropCalendar    必选，定义播种和收获
  ├── TimedEvents     可选，按日历日期触发
  └── StateEvents     可选，按作物状态触发
```

### 9.2 CropCalendar — 作物日历

```yaml
CropCalendar:
    crop_name: 'spring-wheat'
    variety_name: 'spring_wheat_1'
    crop_start_date: 2006-03-31
    crop_start_type: emergence    # sowing=从播种, emergence=从出苗
    crop_end_date: 2006-10-20
    crop_end_type: earliest       # maturity/ harvest/ earliest
    max_duration: 300             # 最长生长期 (保底截止天数)
```

**两种起始方式**：

| start_type | 含义 | 适用场景 |
|-----------|------|---------|
| `sowing` | 从播种开始模拟 | 需要考虑播种到出苗阶段的土壤水分和温度影响 |
| `emergence` | 从出苗开始模拟 | 跳过播种阶段，直接从已有幼苗开始 |

**三种结束方式**：

| end_type | 含义 | 行为 |
|----------|------|------|
| `maturity` | 等生理成熟 | 模型运行至 DVS=2.0 自然停止 |
| `harvest` | 到指定日期强制收 | 到达 crop_end_date 就终止，不管 DVS 多少 |
| `earliest` | 先到为准 | 成熟和指定日期哪个先到就按哪个结束 |

### 9.3 TimedEvents — 定时事件

到指定日期就触发，不管作物长到哪个阶段：

```yaml
TimedEvents:
-   event_signal: apply_n
    events_table:
    - 2006-04-10: {amount: 10, recovery: 0.7}   # 4月10日施氮
    - 2006-05-05: {amount:  5, recovery: 0.7}   # 5月5日追加

-   event_signal: irrigate
    events_table:
    - 2006-07-01: {amount: 20, efficiency: 0.7}
```

**适用**：操作时机由日历决定，如"每年4月10日施底肥""入冬前灌一次防冻水"。

### 9.4 StateEvents — 状态事件

不等到特定日期，等作物或土壤达到某个阈值时触发。使用零交叉检测：

```yaml
StateEvents:
-   event_signal: apply_n
    event_state: DVS              # 监控发育阶段
    zero_condition: rising        # 从下方穿越阈值
    events_table:
    - 0.3: {N_amount: 1, N_recovery: 0.7}     # 分蘖期施氮
    - 0.6: {N_amount: 11, N_recovery: 0.7}    # 拔节期施氮

-   event_signal: irrigate
    event_state: SM               # 监控土壤水分
    zero_condition: falling       # 从上方穿越阈值
    events_table:
    - 0.15: {amount: 2, efficiency: 0.7}      # 干了就浇
```

**零交叉检测**避免阈值附近反复触发。每一步计算 `sign = cmp(current - threshold, 0)`，与上一步对比：

| zero_condition | 触发条件 | 含义 |
|---------------|---------|------|
| `rising` | sign 从 -1 变为 >=0 | 状态从阈值下方穿越到上方 (DVS 0.29→0.30) |
| `falling` | sign 从 1 变为 <=0 | 状态从阈值上方穿越到下方 (SM 0.16→0.15) |
| `either` | 任意方向穿越 | 不管方向，跨过就触发 |

**适用**：操作时机由作物或土壤状态决定——"长到拔节期追肥""干了就浇水"。每年实际日期随气温和降雨变化。

### 9.5 可用操作信号

| 信号 | 操作 | 参数 | 发出方 |
|------|------|------|--------|
| `crop_start` | 开始作物 | crop_name, start_type | CropCalendar 自动 |
| `crop_finish` | 结束作物 | finish_type | CropCalendar 自动 |
| `irrigate` | 灌溉 | amount (cm), efficiency (0-1) | TimedEvents / StateEvents |
| `apply_n` | 施氮肥 | amount (g/m²), recovery (0-1) | TimedEvents / StateEvents |
| `apply_npk` | 施氮磷钾复合肥 | N/P/K_amount | TimedEvents / StateEvents |
| `mowing` | 割草 | biomass_remaining | TimedEvents / StateEvents |
| `terminate` | 终止模拟 | 无 | TimedEvents |

`crop_start` 和 `crop_finish` 由 CropCalendar 内部自动发出，用户写 YAML 时主要在 TimedEvents/StateEvents 中配置 `irrigate`、`apply_n`。

### 9.6 完整示例：两年轮作

```yaml
AgroManagement:
- 1999-08-01:                        # 第一年：冬小麦
    CropCalendar:
        crop_name: wheat
        crop_start_date: 1999-09-15
        crop_start_type: sowing
        crop_end_type: maturity
        max_duration: 300
    TimedEvents:
    - event_signal: irrigate
      events_table:
      - 2000-01-01: {amount: 2}
      - 2000-01-21: {amount: 5}

- 2001-01-01:                        # 休耕 (空活动)
    CropCalendar: null
    TimedEvents: null
    StateEvents: null

- 2001-03-01:                        # 第二年：青贮玉米
    CropCalendar:
        crop_name: maize
        crop_start_date: 2001-04-15
        crop_start_type: sowing
        crop_end_type: maturity
        max_duration: 200
    StateEvents:
    - event_signal: apply_n
      event_state: DVS
      zero_condition: rising
      events_table:
      - 0.3: {N_amount: 1}
      - 0.6: {N_amount: 11}
    - event_signal: irrigate
      event_state: SM
      zero_condition: falling
      events_table:
      - 0.15: {amount: 2}

- 2001-12-15:                        # 尾部空活动 (模拟结束)
```

**关键规则**：
- 活动之间允许空档（休耕期），期间只运行土壤水分平衡
- 每个活动最多一个 CropCalendar，可以有多个 TimedEvents 和 StateEvents
- **如果最后一个活动包含 StateEvents，必须后跟尾部空活动**——因为 StateEvents 触发时间不可预测，没有尾部空活动就无法确定模拟结束日期

### 9.7 TimedEvents vs StateEvents

| 场景 | 用什么 | 原因 |
|------|--------|------|
| 固定日期追肥 | TimedEvents | 日期确定 |
| 按生育期追肥 (拔节期追氮) | StateEvents(DVS) | 发育速度取决于当年气温 |
| 按墒情灌溉 (干了就浇) | StateEvents(SM) | 土壤水分取决于降雨 |
| 固定灌溉计划 (每周浇一次) | TimedEvents | 管理策略决定 |

判断标准：触发时机是由日历决定还是由作物/土壤状态决定。

### 9.8 Demo 数据库中的农事操作

Demo DB 的 `crop_calendar` 表只有最简信息——有 CropCalendar 但 TimedEvents 和 StateEvents 都是 null。`AgroManagementDataProvider` 把它转成：

```yaml
- 2000-01-01:
    CropCalendar:
        crop_name: 'WINTER WHEAT'
        crop_start_date: 2000-01-01
        crop_start_type: emergence
        crop_end_date: null
        crop_end_type: maturity
        max_duration: 365
    TimedEvents: null
    StateEvents: null
```

**所以用 `pcse.start_wofost()` 跑 Demo 时，模型只管模拟作物在给定天气下的自然生长，不干预任何施肥或灌溉。** 需要这些操作必须手写完整的 YAML 文件。

### 9.9 执行流程

模拟过程中每天的执行顺序：

```
AgroManager(day)
  ├── 检查是否该切换到下一个 Campaign
  ├── CropCalendar: 检查播种/收获日期
  ├── TimedEvents: 遍历定时事件，触发当天到期的
  └── StateEvents: 遍历状态事件，检查阈值穿越

Engine(day)
  ├── AgroManager(day)    # 先处理农事操作 (发出信号)
  ├── calc_rates(day)     # 计算今天的生长速率
  └── integrate(day)      # 积分更新状态
```

农事管理器在每一步最先执行——如果今天该播种，先触发 `crop_start` 信号初始化作物状态，然后才进入光合/呼吸/蒸腾的计算。

### 9.10 对硅基一号的启示

| PCSE 概念 | 温室对应 | 备注 |
|-----------|---------|------|
| Campaign | 一茬番茄 (定植→拉秧) | 番茄是无限生长型，没有 DVS=2.0 |
| CropCalendar | 定植日期 + 预期拉秧日期 | 温室可周年生产 |
| TimedEvents(irrigate) | 定时滴灌 | 温室频率更高 (分钟→小时级) |
| StateEvents(SM, irrigate) | 基于基质含水量自动灌溉 | **直接对应**，闭环控制 |
| StateEvents(DVS, apply_n) | 生育阶段营养液配方切换 | PCSE 仅氮，温室需全元素配方 |
| 休耕空活动 | 两茬之间清棚/消毒 | 概念一致 |
| **缺失** | 补光控制 (基于 DLI 或定时) | PCSE 无补光概念 |
| **缺失** | CO2 施肥 | PCSE 无 CO2 模块 |
| **缺失** | 环控 (温/湿/通风设定值切换) | PCSE 无环控概念 |

最有价值的设计遗产是 **StateEvents 的零交叉检测机制**。对温室闭环控制天然适用——"当基质含水量低于 60% 开启滴灌""当 DLI 低于 10 时开启补光"。硅基一号的控制策略层可以直接沿用这个模式，只需扩展新的 `event_state` (基质 EC、pH、CO2 浓度等) 和新的 `event_signal` (补光、通风、环控设定值切换等)。
